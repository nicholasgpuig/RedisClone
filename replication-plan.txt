 The one wrinkle: commands can wrap around the end of the ring. You either handle that in the parser (awkward), or when you detect that a command would straddle the end, you memmove the partial command to the front before the next recv(). That's far cheaper than the current approach which memmoves the entire remaining buffer on every command.

Yes, you need two levels of locking:

1. A readers-writer lock (or mutex) on the map itself — held shared (read) when appending to an existing replica's buffer, held exclusive (write) when adding or removing a replica from the map
2. A per-replica mutex on each buffer — held briefly while appending command bytes

The sequence for an epoll thread propagating a write:
shared_lock(map_mutex)        // many epoll threads can hold this simultaneously
for each replica in map:
    lock(replica.buf_mutex)   // brief, just the append
    append command bytes
    unlock(replica.buf_mutex)
shared_unlock(map_mutex)

Adding/removing a replica (rare, on connect/disconnect):
exclusive_lock(map_mutex)     // blocks all epoll threads briefly
insert or erase from map
exclusive_unlock(map_mutex)

std::shared_mutex with std::shared_lock / std::unique_lock is the stdlib type for this. The outer shared lock is held for almost no time (just long enough to iterate and kick off per-buffer appends), so the exclusive lock for add/remove will rarely block epoll threads for long.

The key insight: readers-writer lock is worth it here because replica add/remove is extremely rare compared to the per-command propagation writes. A plain mutex would serialize all epoll threads on every command even though they could all safely read the map simultaneously.




Track a monotonically increasing byte counter on both sides:

Master side: maintain std::atomic<uint64_t> repl_offset in Storage. Every time a write command is propagated to replicas, increment it by the byte length of the RESP command. This happens in send_to_replicas.

Replica side: maintain a local uint64_t repl_offset. Every time the replica applies a command received from the master, increment it by the byte length of that command (which you already know from state.start_idx in the parse loop).

Handshake change: instead of PSYNC ? -1, the replica sends PSYNC <replication_id> <offset>. Master compares the replica's offset against its replication backlog. If the delta is within the backlog, partial resync — send just the missing bytes. If not, full resync.

Replication backlog: a fixed-size ring buffer (say 1MB) on the master that holds the last N bytes of propagated commands. This is what makes partial resync possible — instead of re-sending the full snapshot, the master just sends backlog[replica_offset..master_offset].

For failover specifically, the offset is simpler — you just need replicas to report their offset during election so the most up-to-date one can be elected. You don't need partial resync working for that. So the minimal implementation is:

1. Increment a counter on master per propagated byte
2. Increment a counter on replica per applied byte
3. Replica reports its offset in the SUSPECT / election messages
4. Highest offset wins election

The replication backlog (for partial resync after reconnect) is a separate optimization you can defer.
